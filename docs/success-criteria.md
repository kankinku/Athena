# 성공 기준 (Success Criteria)

이 문서는 Athena 모듈 협의 시스템의 **측정 가능한 성공 기준**을 정의한다.
모든 기준은 검증 가능한 형태로 명시하며, 파일럿 적용(Task 15) 시 실제로 측정한다.

---

## 핵심 성공 기준

### SC-01: 자동 영향 모듈 식별

**기준**: 변경된 파일 목록이 주어졌을 때, 모듈 의존 그래프를 기반으로 영향 받는 모듈이 자동으로 식별된다.

**측정 방법**:
```bash
athena impact --changed-paths src/store/migrations.ts
# Expected output:
# Direct:   store (owner: store-agent)
# Indirect: research, cli, ui (depend on store)
# Observer: remote (uses store indirectly)
```

**합격 기준**: 3개 테스트 시나리오에서 영향 모듈 식별 정확도 100% (알려진 의존 관계 기준).

---

### SC-02: 관련 에이전트만 소집

**기준**: 변경의 영향도에 따라 필요한 에이전트만 소집되고, 불필요한 에이전트는 소집되지 않는다.

**측정 방법**:
- 단일 모듈 내부 변경 → 해당 모듈 오너 1개만 소집됨
- 공용 인터페이스 변경 → 의존 모듈 오너 모두 소집됨
- DB 스키마 변경 → store + research + cli 소집됨

**합격 기준**:
- 불필요 소집 없음 (정밀도 100%)
- 필요 소집 누락 없음 (재현율 100%)
- 3개 시나리오 모두 합격

---

### SC-03: 회의 결과 → 실행 계획 변환

**기준**: 에이전트 회의 합의 결과가 구조화된 실행 계획(`approved_plan`)으로 변환된다.

**측정 방법**:
```bash
athena proposal status <proposal-id>
# Must show: meeting_state=agreed, execution_plan=present, assigned_agents=list
```

**합격 기준**:
- 합의 도달 후 실행 계획 자동 생성됨
- 실행 계획에 `task_assignments`, `required_tests`, `rollback_plan` 모두 존재
- 실행 계획 없이 실행 시작 불가능

---

### SC-04: 통합 테스트 실패 → 재협의 자동 트리거

**기준**: 실행 후 통합 테스트가 실패하면 재협의(`remeeting`)가 자동으로 트리거된다.

**측정 방법**:
1. change proposal 생성 → 합의 → 실행
2. 시뮬레이션된 테스트 실패 주입
3. 시스템 상태 확인: `workflow_state = remeeting`

**합격 기준**:
- 테스트 실패 후 5분 이내 `remeeting` 상태로 전환
- 재협의에 원래 회의 참석자 + 실패 모듈 오너가 소집됨
- 재협의 결과가 원래 회의 기록에 연결됨

---

## 보조 성공 기준

### SC-05: 회의 품질

- 평균 회의 라운드 수 ≤ 5 (합의 도달 시)
- 에이전트 미응답 처리: 타임아웃 5분 이내 적용
- 충돌 해결률 ≥ 80% (재회의 없이)

### SC-06: 실행 안전장치

- 승인 없는 공용 인터페이스 변경: 100% 차단
- 승인 없는 DB 스키마 변경: 100% 차단
- 모든 실행에 롤백 계획 존재

### SC-07: 감사 가능성

- 모든 소집, 발언, 결정, 실행 단계가 감사 로그에 기록됨
- 과거 회의 전체 재현 가능 (타임스탬프 + 에이전트 발언 포함)

### SC-08: 성능

- 영향도 분석 완료: < 10초 (100개 파일 변경 기준)
- 에이전트 소집 완료: < 30초 (5개 모듈 기준)
- 회의 진행: < 15분 (일반 변경 기준)

---

## 실패 조건 (즉시 중단)

다음 조건이 발생하면 파일럿을 중단하고 설계를 재검토한다:

1. 운영자 인지 없이 공용 인터페이스가 변경됨
2. 롤백 계획 없는 실행이 시작됨
3. 재협의 없이 실패한 테스트가 무시됨
4. 에이전트 소집 기록이 DB에 남지 않음
