# 현재 상태 매핑

이 문서는 Athena가 이미 가지고 있는 구성요소를 `자율 연구 시스템` 관점에서 다시 매핑한다.

## 현재 Athena의 실제 중심

Athena는 이미 다음 요소를 갖고 있다.

- 목표를 장기간 추적하는 연구 런
- 제안, 결정, 실험, 보고를 저장하는 구조화된 상태
- 자동화 정책과 재시도, 체크포인트, 복구
- 로컬 및 원격 실행
- 메트릭 수집과 결과 비교
- 운영자 관찰 및 승인 표면

즉, Athena는 핵심 없는 도구가 아니라 `루프형 시스템의 뼈대`를 이미 가지고 있다.

## 기존 구성요소의 올바른 해석

| 구성요소 | 지금의 올바른 의미 |
|---|---|
| `ProposalBrief` | 다음 개선 후보 |
| `DecisionRecord` | 개선 방향 선택 기록 |
| `SimulationRunner` | 평가 단계 실행 엔진 |
| `AutomationManager` | 루프 지속과 복구 관리자 |
| `TeamOrchestrator` | 방향 제어기 |
| `Claim` / evidence | 개선 근거 계층 |
| `reporting.ts` | 루프 설명 계층 |

## 유지할 것

- `src/research/*`의 상태 모델
- `src/remote/*`의 실행 인프라
- `src/security/*`의 정책 계층
- `src/memory/*`의 그래프 메모리
- `src/cli/*`, `src/ui/*`의 운영 표면

## 다시 정의할 것

- proposal은 "회의용 change request"가 아니라 "다음 개선 후보"
- operator surface는 제품 본체가 아니라 관찰과 개입 수단
- autonomy roadmap은 별도 시스템이 아니라 같은 루프의 확장 경로
- change-management와 meeting 계층은 연구 루프에 직접 통합되기 전까지 제품 중심을 정의하지 않아야 한다
- 지원되지 않은 실험 표면은 루트 제품 표면보다 앞에 오면 안 된다

## 핵심 진단

현재 문서의 가장 큰 문제는 `루프 중심 설명`보다 `보조 메커니즘 중심 설명`이 앞선다는 점이다.

정확한 설명 순서는 다음이어야 한다.

1. 목표를 향한 반복 개선 루프
2. 연구와 증거 수집
3. 실행과 평가
4. 오케스트레이션과 메모리
5. 운영자 감독, 정책, 승인
