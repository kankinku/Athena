# 모듈 오너십 문서 (Module Ownership)

이 문서는 `config/module-registry.yaml`에 정의된 모듈 오너십의 **책임 범위와 경계 규칙**을 사람이 읽기 좋은 형태로 설명한다.

---

## 오너십 원칙

### 원칙 1: 모든 코드 경로에 오너가 있다

Athena 코드베이스의 모든 소스 파일은 정확히 하나의 모듈에 속하며, 해당 모듈에 담당 에이전트가 배정된다.

### 원칙 2: 오너는 자신의 범위에서 권한을 갖는다

모듈 오너 에이전트는:
- 자신의 모듈 내부 변경을 독립적으로 실행할 수 있다 (정책 허용 범위 내)
- 자신의 공용 인터페이스 변경을 제안할 수 있다
- 다른 모듈의 자신 인터페이스 사용 방식에 대해 의견을 제시할 수 있다

### 원칙 3: 범위 외 변경에는 협의가 필요하다

에이전트가 자신의 모듈 범위 외 코드를 수정하려면 반드시:
1. change proposal을 통해 의도를 명시하고
2. 영향받는 모듈 오너들과 회의를 통해 합의를 얻어야 한다

---

## 모듈별 책임 범위

### `store` 모듈 — store-agent

**수정 가능**:
- `src/store/database.ts` — DB 연결 로직
- `src/store/migrations.ts` — 새 마이그레이션 추가 (단, v번호 순서 준수)
- `src/store/session-store.ts` — 세션 CRUD
- `src/store/preferences.ts` — 사용자 설정

**승인 없이 수정 불가**:
- 기존 마이그레이션 SQL 변경 (데이터 손실 위험)
- DB 파일 경로/이름 변경
- 기존 테이블/컬럼 삭제 또는 이름 변경

**다른 모듈 협의가 필요한 경우**:
- 기존 테이블에 NOT NULL 컬럼 추가 (모든 모듈의 INSERT 쿼리 영향)
- `getDb()` API 변경 (모든 모듈이 사용)

---

### `research` 모듈 — research-agent

**수정 가능**:
- 새 store 클래스 추가 (내부 사용)
- 워크플로 상태 전환 로직
- 제안 스코어링 로직 (내부 알고리즘)
- 자동화 정책 로직

**승인 없이 수정 불가**:
- `contracts.ts` 공용 타입 변경 (cli, ui 모두 영향)
- `WorkflowState` enum 값 추가/제거
- 새 DB 테이블 추가 (store 모듈 협의 필요)

**다른 모듈 협의가 필요한 경우**:
- `contracts.ts`의 공용 인터페이스 변경 → cli, ui 동시 협의
- 새 DB 테이블 → store 협의
- 새 감사 이벤트 → security 협의

---

### `impact` 모듈 — impact-agent

**수정 가능**:
- 영향도 분석 알고리즘 내부 로직
- 그래프 빌더 최적화
- 분석 결과 캐시 전략

**승인 없이 수정 불가**:
- `config/module-registry.yaml` — 모든 모듈에 영향
- 소집 규칙 변경 — 회의 참석자 구성 변경

**다른 모듈 협의가 필요한 경우**:
- 모듈 경계 재정의 → 해당 모듈 협의
- 새 위험도 레벨 추가 → 모든 모듈 협의

---

### `cli` 모듈 — cli-agent

**수정 가능**:
- 새 CLI 커맨드 추가 (기존 커맨드에 영향 없을 때)
- 출력 형식 개선 (하위 호환)
- 에러 메시지 개선

**승인 없이 수정 불가**:
- 기존 커맨드 인터페이스 변경 (사용자 영향)
- 커맨드 제거

**다른 모듈 협의가 필요한 경우**:
- 새 research 커맨드 추가 → research 협의
- 새 보안 커맨드 → security 협의

---

### `ui` 모듈 — ui-agent

**수정 가능**:
- 패널 내부 렌더링 로직
- 테마/색상 조정
- 새 패널 추가 (layout.tsx 변경 없이)

**승인 없이 수정 불가**:
- `layout.tsx`의 패널 배치 구조 변경
- 전역 키 바인딩 변경

**다른 모듈 협의가 필요한 경우**:
- research 데이터 모델 변경에 따른 패널 업데이트 → research 협의

---

### `remote` 모듈 — remote-agent

**수정 가능**:
- SSH 연결 파라미터 조정
- 오류 처리 개선
- 플랫폼별 경로 처리

**승인 없이 수정 불가**:
- 원격 실행 격리 수준 낮추기 (보안 위험)
- SSH 키 처리 로직 변경

**다른 모듈 협의가 필요한 경우**:
- 실행 결과 형식 변경 → tools, research 협의
- 보안 정책 우회 → security 협의 (항상 금지)

---

### `security` 모듈 — security-agent

**수정 가능**:
- 위험 패턴 추가 (강화 방향)
- 감사 로그 세부 항목 추가
- 오류 메시지 개선

**승인 없이 수정 불가**:
- 기존 차단 패턴 제거 또는 완화
- enforce → audit 모드 전환
- 보호 경로 제거

**다른 모듈 협의가 필요한 경우**:
- 새 커맨드 패밀리 분류 → tools, remote 협의
- 정책 완화 → operator 필수 승인

---

### `tools` 모듈 — tools-agent

**수정 가능**:
- 도구 내부 구현 로직 개선
- 새 편의 도구 추가 (보안 정책 범위 내)

**승인 없이 수정 불가**:
- 도구 인터페이스(파라미터/반환값) 변경
- 새 보안 민감 도구 추가

**다른 모듈 협의가 필요한 경우**:
- 원격 실행 도구 변경 → remote 협의
- 보안 민감 도구 → security 협의

---

### `providers` 모듈 — providers-agent

**수정 가능**:
- 새 모델 ID 추가
- 응답 파싱 로직 개선
- 재시도 로직 조정

**승인 없이 수정 불가**:
- 인증 토큰 저장 경로 변경
- OAuth 플로우 변경

**다른 모듈 협의가 필요한 경우**:
- 새 AI 공급자 추가 → tools, research 협의
- 인증 API 변경 → store 협의

---

## 경계 분쟁 해결 절차

모듈 경계가 불명확할 때 (예: "이 파일이 어느 모듈에 속하는가?"):

1. `config/module-registry.yaml`의 `paths` 패턴이 최종 기준
2. 패턴 충돌 시 impact-agent가 판정
3. 판정이 어려우면 운영자가 결정하고 registry 업데이트

---

## 레지스트리 업데이트 절차

`config/module-registry.yaml`을 변경하려면:

1. change proposal 생성 (타겟 모듈: `impact`)
2. 모든 영향받는 모듈 오너에게 알림
3. 운영자 최종 승인
4. 업데이트 후 영향도 분석 재실행
