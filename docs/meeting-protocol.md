# 협의 프로토콜

이 문서는 다중 모듈 조정이 필요할 때 Athena가 사용하는 협의 절차를 정의한다.

## 언제 필요한가

협의는 항상 필요한 것이 아니다.

다음 경우에만 필요하다.

- 여러 모듈이 동시에 영향을 받는 경우
- 공용 계약 변경이 포함된 경우
- 보안, 데이터, 배포 위험이 높은 경우
- 단일 루프 안에서 해결할 수 없는 충돌이 생긴 경우

## 목표

협의의 목적은 토론 자체가 아니라 `다음 실행을 더 정확하고 안전하게 정렬하는 것`이다.

## 기본 흐름

1. 현재 목표와 변경 의도를 공유한다.
2. 각 모듈 또는 역할이 영향을 평가한다.
3. 충돌과 조건을 정리한다.
4. 실행 가능한 다음 계획을 확정한다.
5. 검증 조건과 중단 조건을 함께 기록한다.

## 중요한 원칙

- 협의는 루프를 멈추게 하기 위한 장치가 아니다.
- 협의는 루프가 잘못된 방향으로 달리지 않게 하기 위한 정렬 장치다.
- 해결되지 않은 충돌은 보류하거나 상위 승인을 받는다.
